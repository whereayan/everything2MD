# 1 Introduction

This guide was developed by the CDISC Submission Data Standards (SDS) Study Data Tabulation Model (SDTM) Conformance Rules subteam to explain the methodology used to create SDTM conformance rules and the logic applied to determine which implementation guidance should be translated into rules.

## 1.1 Purpose

The purpose of this guide is to provide methodology for the identification and classification of implementation text that constitutes a rule definition.

## 1.2 Scope

SDTM conformance rules have an explicit basis in text from the SDTM document, the SDTM Implementation Guide (IG), and the SDTMIG: Associated Persons.

The following are considered to be outside the scope of SDTM conformance rules:

- Rules that are not supported by a definitive statement in the underlying documents

- Business rules

- Data validation logic that is not specifically linked to SDTM implementation guidance

SDTM conformance rules are identified and classified regardless of the perceived programmability of the rule.

For each identified SDTM conformance rule, a logic statement is provided, consisting of scoping attributes (e.g., applicable domains); the rule itself; and, if applicable, a conditional statement.  Unless otherwise specified, it is implicit in rule logic that it applies to a given subject or record.

**Appendix A: Definitions of Rule Sets**

There are several types of rule sets surrounding SDTM. These rules can be divided into 4 categories:

1. Conformance Rules: Conformance rules are created and maintained by CDISC. Conformance rules describe the criteria that must be met to be in compliance with the CDISC standard.

2. Business Rules: Business rules are created by a specific organization to describe the criteria that should be met to allow for the deliverable to be useful in the conduct of normal business practices. For example, US Food and Drug Administration (FDA) Business Rules describe criteria that should be met in order for datasets to be utilized internally for FDA business practices such as submission review.

3. Technical Rejection Criteria for Study Data: eCTD Technical Rejection Criteria describe the minimum requirements for eCTD submissions to be accepted by the agency at the gateway.

4. Validator Rules: Validator rules are rule sets utilized by validation tools. Each validator can have its own set of validation rules. Following are some examples of validator rule sets; others are also available:

    a. FDA validator rules describe the validation rules used by the FDA.

    b. Pinnacle 21 validator rules describe the validation rules used by the Pinnacle 21 validator tool.

    c. PointCross' eDataValidator

validator rules describe the validation rules used by PointCross' eDataValidator.
